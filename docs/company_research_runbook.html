Company Research Runbook

This runbook explains how to rebuild, validate, inspect, and package the company research outputs without rebuilding the repository or creating a second architecture.

Rerun Process

  1. Update scripts/company_research/company_research_data.py with the source set, entity definitions, people, and relationship candidates.
  2. Run python scripts/company_research/run_company_research_production.py --group venuiti.
  3. For PowerShell users, run powershell -ExecutionPolicy Bypass -File scripts/company_research/run_company_research_production.ps1.
  4. Inspect the generated files under data/company_research/venuiti/ and the package under .ai_downloads/.

Validation Steps

  • The build script rewrites the shared schema and validates company_people_research.json with Draft 2020-12.
  • scripts/company_research/validate_company_research_outputs.py validates required outputs, inferred graph references, allowed relationship labels, package manifest alignment, and zip contents.
  • tests/run_all.py includes the company research validator so pull-request validation catches drift before release.
  • Python files should pass py_compile.
  • Review run_company_research_production.log.txt, build_company_people_research.log.txt, render_company_research_visuals.log.txt, and package_company_research.log.txt after each run.
  • If a relationship score drops below acceptance thresholds, preserve that uncertainty instead of forcing a merge.

Updating Evidence

  • Add new official sources before relying on secondary sources.
  • Use secondary sources for corroboration, employee overlap, founding year, or headquarters support.
  • Do not convert group-language into legal-ownership language unless a primary legal source supports it.

Production Controls

  • The GitHub validation workflow installs Python dependencies from requirements.txt and runs tests/run_all.py.
  • The GitHub package workflow rebuilds the Venuiti company research package before creating release artifacts.
  • The production runner fails fast on any non-zero build, render, validation, package, or repository validation step.
  • The validator rejects broken inferred-graph references, unsupported relationship labels, missing evidence, and package/manifest mismatch.

Failure Handling

If the production runner fails, open data/company_research/venuiti/run_company_research_production_summary.json first to identify the failed step. Correct schema mismatches or malformed curated data before re-running. If packaging fails, verify the normalized outputs, docs, and package manifest exist before re-running packaging. All production scripts return non-zero exit codes for deterministic automation behavior.
