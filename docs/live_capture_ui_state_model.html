Live Capture UI State Model

This model defines how the shared viewer panel in my_git_documents reacts to bridge availability, run state changes, page refresh, and delayed outputs.

Refresh and Recovery

  • On page load, the panel requests GET /api/capture/status and GET /api/runs.
  • If the status response includes a current run_id, the UI then requests GET /api/runs/{run_id} for richer details.
  • This allows the panel to survive a browser refresh and rebuild the active run view from bridge state rather than browser memory.
  • The elapsed timer rehydrates from duration_seconds and started_at and continues locally while the run remains active.

Warning and Failure Rendering

  • Warnings returned by the bridge are shown in a visible warnings panel and do not block the rest of the UI.
  • Transport failures render separately from run-state failures so the user can tell whether the bridge is down or the run itself failed.
  • warn and fail states still render available outputs and history.
  • Missing fields are rendered as not ready, none, or not provided rather than causing exceptions.

Display Rules

UI element Rule
Status badge Reflects the normalized run state or offline for transport failure.
Current Run ID Shows the active run when available, otherwise none.
Result links Enabled only when the bridge can provide a browser-safe URL or artifact route fallback.
Transcript preview Attempts to fetch the transcript text through a browser-safe link. If unavailable, the panel explains why instead of trying local filesystem access.
Recent runs Shows the most recent runs from GET /api/runs, with quick links and a per-run load action.
Prep context Loads and summarizes the interview data inputs from career, story, company research, and opportunity JSON sources configured for the workspace.
